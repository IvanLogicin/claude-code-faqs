Welcome Screen Customization

When you run claude in the terminal, a welcome panel appears showing your name, recent activity, and getting-started tips.

Not currently customizable. The welcome screen is generated by the Claude Code application itself and is not driven by any locally editable config file. To request this feature, file an issue at github.com/anthropics/claude-code/issues.

What Is Configurable

  • Status line (the bar shown during a session) — use the /statusline command
  • Theme, model, effort level — via /config
  • Permissions, hooks, env vars — via ~/.claude/settings.json or project-level .claude/settings.json
  • Keyboard shortcuts — via ~/.claude/keybindings.json

Thinking Mode & Effort Levels

Thinking mode gives Claude extra compute to reason through a problem step-by-step before responding (visible as a collapsible "thinking" block). It uses the same model — no model switch occurs.

What you gain with thinking ON

  • Deeper reasoning on complex problems
  • Better architecture & logic decisions
  • More reliable multi-step answers

What you gain with thinking OFF

  • Faster responses
  • Lower token usage
  • Better for simple, quick tasks
Turning thinking off has zero effect on Claude's ability to read project files or documentation. File access is handled by separate tools, not the reasoning system.

How to Control It

MethodAction
Alt+TToggle thinking on/off during a session
/effort commandSet a specific level: off, low, medium, high, max
Defaultauto (currently resolves to medium)

Switching Mid-Session

The warning shown when toggling is a heads-up about latency, not a sign of a real problem. Switching mid-session is safe — it affects how the next message is processed, not past context.

Parallel Instances

Running multiple claude terminals simultaneously is fully supported. Each session is independent with its own context. This is useful for genuinely parallel workflows, but not necessary just to manage thinking mode.

Cloud Processing vs Local Resources

How It Works

All AI processing happens on Anthropic's servers. Your laptop only runs the Claude Code CLI (a lightweight Node.js app) which sends messages to the API and displays responses.

What This Means Practically

  • Your laptop's CPU, RAM, and GPU have no effect on response speed or quality
  • Switching to a faster PC will not improve response speed
  • Speed is determined by: network latency + Anthropic server load + thinking budget allocated
  • A faster internet connection may help marginally

File Format Support & Limits

PDF Files

Claude reads PDFs visually (as rendered page images), not by extracting raw text. This means:

  • Non-OCR'd / scanned PDFs: Readable — Claude sees the page like a human would
  • Bad OCR / garbled text layer: Not an issue — Claude reads the visual rendering, not the broken text
  • Limit: 20 pages per request. Large PDFs must be read in chunks using page ranges

Plain Text & CSV

Read line-by-line. Default is 2,000 lines per read. Larger files can be paginated using offset and limit parameters. No hard file size cap — just pagination.

Word (.docx) & Excel (.xlsx)

Native support is uncertain — these are binary formats and may not be directly readable by the Read tool. Workarounds include converting them via Bash (e.g., using libreoffice --headless, python-docx, or openpyxl). To be verified with live testing.

Google Docs Access

Claude Code can access Google Docs via the WebFetch tool, but accessibility depends on how the document is shared.

Access by Share Type

SourceAccessible?Content QualityNotes
Private Google Doc No — Requires login, blocked entirely
"Anyone with link" (raw URL) Unreliable Messy Google Docs is a JS app; raw fetch returns incomplete HTML
"Anyone with link" + export URL Yes Clean Append /export?format=txt or /export?format=pdf to the doc URL
Published to Web (/pub URL) Yes Clean Renders as static HTML, best option for sharing with Claude

Recommended Approaches

  1. Publish to Web (File → Share → Publish to Web) — cleanest, most reliable
  2. Export URL trick for publicly shared docs — no republishing needed:
    • Plain text: https://docs.google.com/document/d/[DOC_ID]/export?format=txt
    • PDF: https://docs.google.com/document/d/[DOC_ID]/export?format=pdf
The same access patterns apply to Google Sheets — export as format=csv or format=xlsx from a publicly shared sheet.

Sessions & Context — How They Work

Every conversation you have with Claude Code is a session. Sessions are saved automatically and can be resumed later, even after closing the terminal or rebooting.

Where Sessions Are Stored

Everything is stored locally on your machine — nothing conversation-related is sent to or stored in Anthropic's cloud.

PathContents
~/.claude/sessions/General session metadata
~/.claude/projects/[folder-path]/Per-project conversation logs, one .jsonl file per session
~/.claude/history.jsonlYour command/input history across all sessions

The project subfolder name is derived from your working directory path (e.g., -home-ivan-Documents-my-project). Your actual project folder only ever contains files you explicitly create.

File Format

Sessions are stored as .jsonl files (JSON Lines) — one JSON object per line, each representing a message or event in the conversation. They are plain text and can be opened in any text editor. Each file is named with a UUID (e.g., 23f012d1-0f95-48d2-8258-60017da99dfd.jsonl).

How to Resume a Session

  1. Open a terminal and navigate to your project folder (important — so the right CLAUDE.md loads)
  2. Run claude
  3. Type /resume — a list of recent sessions appears, sorted by date
  4. Use ↑ / ↓ arrow keys to select the session, then press Enter
  5. The full conversation is re-rendered in the terminal and you can scroll back through all of it
If a session doesn't appear in the /resume list (e.g., it's older), you can find it manually by running ls -lht ~/.claude/projects/[your-folder]/ to list sessions by date, then pass the UUID directly to /resume.

Auto-Deletion Policy

There is no documented auto-expiry for sessions. Files persist until you delete them manually. Sessions are small (a long conversation is typically only a few MB), so disk space is rarely a concern. To manually remove a session, delete its .jsonl file from the project folder under ~/.claude/projects/.

Naming & Renaming Sessions

By default, sessions receive an auto-generated name. You can assign custom names for easier identification and resumption:

MethodCommandNotes
Name at launchclaude -n "session-name"Sets the name before the session starts
Rename mid-session/renameAlso updates the name shown in the prompt bar
Resume by nameclaude --resume "session-name"Skips the interactive picker entirely

Suggested naming convention: project-topic-date — e.g., logicinbound-migration-2026-04-13. Consistent names make sessions easy to find in the /resume picker without scrolling.

CLAUDE.md — Project & Global Instructions

CLAUDE.md is a plain Markdown file that Claude Code reads automatically at the start of every session. It lets you give Claude standing instructions, project context, and preferences without repeating yourself each time.

Global vs Project-Specific

TypeLocationScope
Global ~/.claude/CLAUDE.md Loaded in every Claude Code session, regardless of folder
Project-specific [project-folder]/CLAUDE.md Loaded only when Claude Code is started from that folder

Both files are loaded together when present — global first, then project-specific on top. Project instructions override or extend global ones.

How to Create One

Just create the file — Claude Code picks it up automatically on the next session start. No configuration required.

  • Global: create ~/.claude/CLAUDE.md in your home directory
  • Project: create CLAUDE.md in the root of your project folder (where you run claude from)
  • You can also ask Claude to write or update it for you during a session

What to Put In It

Good candidates for CLAUDE.md content:

  • Tone & style rules — "be concise", "avoid bullet points unless asked", "use formal language"
  • Project conventions — naming patterns, versioning rules, folder structure notes
  • Standing preferences — "never overwrite previous file versions", "always confirm before deleting"
  • Background context — what the project is, who the audience is, current state of work
  • Recurring instructions — things you find yourself repeating every session

How Much Is Too Much

Keep it focused and scannable. A CLAUDE.md that is too long becomes noise — Claude reads all of it, but dense walls of text dilute the signal. Practical guidelines:

  • Aim for under ~100 lines for a project file; under ~50 for a global file
  • Prefer bullet points and short sentences over paragraphs
  • Only include things that are non-obvious or that Claude wouldn't infer from the code itself
  • Remove instructions that no longer apply — stale context causes confusion
  • Don't duplicate things already visible in the codebase (file names, obvious conventions)
CLAUDE.md is just a plain text Markdown file — you can edit it at any time in any text editor. Changes take effect at the start of the next session (or if Claude re-reads it mid-session using the Read tool).

Overriding Global Rules in a Project File

When a project CLAUDE.md covers the same category as the global file, the project version takes precedence. To make the intent explicit, mark the section header with (overrides global) — e.g., ## Tone (overrides global). For single-line overrides, build the exception into the rule itself rather than repeating the full section. This avoids redundancy while keeping the override unambiguous.

Example: Global CLAUDE.md

The following is a real-world global CLAUDE.md file (~/.claude/CLAUDE.md) compiled during a team onboarding session. It covers safety guardrails, tone, behavior, and coding standards suitable for use across all projects on a Linux machine:

# Global Claude Code Instructions

## Local Operations & Safety

- Never modify files or folders before backing them up in non-git projects. In git projects, treat version control as the backup.
- Never delete files. Instead, make a list of files to be deleted as a plain .txt file and present it for review.
- Never run `rm -rf` under any circumstances.
- Never run `rm` on a directory, even with `-r` or `-R` flags.
- Never use the `-f` (force) flag on any `rm`, `git`, or `mv` command.
- Never run `dd`, `mkfs`, `fdisk`, or any disk-level commands.
- Never run `chmod` or `chown` on files outside the current working directory.
- Never read, display, log, or include the contents of .env files, credential files, or any file containing API keys, tokens, or passwords.
- Never commit or write secrets to any file that is tracked by git.
- If a task requires a secret or credential, ask me to provide it directly rather than reading it from a file.

## Tone

- Unless overridden by project-specific Tone instructions, always lean towards concise answers but not so extreme as laconic speak.
- During collaborative planning, be more verbose when detail aids understanding.
- Skip flattery and compliments unless explicitly asked for a judgement.
- Use short affirmations like "Yes", "Yep", and "True" instead of longer ones like "Correct", "Exactly", "Precisely", "You are right", "You've got it", and similar.

## Behavior

- Criticism is welcome. Ask sanity-check questions if something seems impossible or impractical.
- Suggest an alternative approach if it would use fewer tokens, is simpler, is less risky, or produces a better result.
- If in doubt or hitting a snag, suggest an alternative way to proceed and wait for confirmation before continuing.
- When in thinking mode and performing or planning a task, think in terms of first principles while planning out steps ahead.

## Coding

- Reuse existing code: always check for existing functions and constants before creating new ones. Read files thoroughly before writing new functions.
- Maintain consistent naming: follow established patterns throughout the codebase.
- Study the codebase first: thoroughly examine existing patterns before contributing.

Safety Guardrails & Auto-Approve Mode

Claude Code can execute bash commands, edit files, and navigate your entire filesystem — anywhere your OS user has access. This makes it powerful, but also means mistakes can have real consequences. Understanding permissions and guardrails is essential before doing serious work.

What Is Auto-Approve Mode

By default, Claude Code prompts you for confirmation before running bash commands and other potentially impactful actions. In auto-approve mode, it skips those prompts and runs commands immediately without review.

Most reported incidents — including accidental rm -rf deletions — happened in auto-approve mode combined with vague instructions like "clean up the project". Claude executed what it inferred was correct, without a human review step to catch the mistake.

How to Check Your Mode

Check your settings file:

cat ~/.claude/settings.json

An empty result ({}) means you are on defaults — auto-approve is OFF, which is the safe state. Always review bash commands in the confirmation prompt before approving, regardless of what your CLAUDE.md says — instructions are guidance, not a hard technical lock.

Recommended CLAUDE.md Guardrails (Linux)

These rules belong in your global ~/.claude/CLAUDE.md to prevent the most dangerous classes of mistakes:

  • Never run rm -rf under any circumstances
  • Never run rm on a directory, even with -r or -R flags
  • Never use the -f (force) flag on rm, git, or mv
  • Never run dd, mkfs, fdisk, or any disk-level commands
  • Never run chmod or chown on files outside the working directory
  • Never read, display, or log .env files or credential files
  • Never delete files — instead produce a list of candidates for the user to review

Portability Across Operating Systems

The logic behind these guardrails is universal, but the specific commands and paths are OS-dependent:

Linux

  • Guardrails above apply as-is
  • Protect: /etc, /usr, /bin, /boot, /var
  • Dangerous: rm, dd, chmod, fdisk

macOS

  • Same commands apply (Unix base)
  • Also protect: /System, /Library, /Applications
  • Tone/Behavior/Coding sections portable as-is

Windows

  • Safety section needs a full rewrite
  • Dangerous: del /f, rd /s /q, PowerShell Remove-Item -Recurse
  • Protect: C:\Windows, C:\Program Files
  • Tone/Behavior/Coding sections portable as-is

If you share a CLAUDE.md file with teammates on different OSes, the Tone, Behavior, and Coding sections transfer directly. The Safety section should be treated as OS-specific and rewritten per platform.
